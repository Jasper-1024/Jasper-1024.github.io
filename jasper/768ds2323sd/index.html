block-link-plus 开发备忘 & 小记

发表于 2024-07-16 更新于 2025-07-13 分类于 工具 阅读次数： Disqus： 本文字数： 4.6k 阅读时长 ≈ 4 分钟

  • block-link-plus 开发和提交过程的一些记录

  • 更新

导语

个人的第一个 obsidian 插件 block-link-plus 几经波折终于完了, 此文是一些记录和备忘,非完整教程.

  • 据 log 统计,开发思路中断了 x3, 历时 8 个月 (集中开发就周末几天), 目前于官方上架流程中; 官方已合入，
  • 当前 v1.7.5
    • Add plugin: Block Link Plus
  • 现已演进为复杂多功能系统：多行块引用、时间线聚合、内联编辑、流式编辑器等.

Obsidian 的块引用 不支持跨行引用, 个人又非常容易写废话…就挺 ≧ ﹏ ≦ 开始各种寻找解决方案.

历程

Block Reference for multiple paragraph 多行/段 的官方论坛的讨论

  • 跨度从 Oct 2020 一直到今年 Mar 28, 说明这个需求一直存在的.
  • 但是这个 FR 也一直开着, 可能官方暂时不会有这方面动作了.

原初 0.1: 心很大

  • 干脆拓展官方语法: ˅ Unicode 02C5 极少重复.在两个 link 中间的行为一个 跨行块.
  • 空格 or 首行 + ˅link/^link + 空格 or 换行 | 都在行的结尾
  • 这样的好处是不影响 obsidian 现有块跳转, 但是为何官方/其他开发者为何不写呢? 你猜猜? /笑/, 不管你猜不猜, 当时我反正没猜😥

一个猛子扎进了深渊, 可行性评估完全没做. 于是浪费了大量时间, 最终结论是: 现在自己业余 TS 开发做不了.

  • 更深入分析可以参考: Markdown 实现块级引用双向链接的探索
  • 拓展新语法, 修改解析器涉及到了从 MD 文件解析得到 AST 再到 HTML, 这中间那个都不是业余玩家能短期入手的.

教训 1: 天马行空的想法落地, 可行性调查先行.

  • 同类插件, 同类产品
  • 可能的成本, 业余玩家能够投入的成本是否足够.

原初 0.2: 纯添加新 heading 如何?

  • 使用 [file#˅id] ## ˅id ## ^id 整体作为 title
  • 通过分析选中文本段 [start_line,end_lien] 前后和段中的 heading 层级,添加合适的 heading 层级, 是不是就能做到多段的引用了 😊, 为自己的想法点赞!

🧠热了, 乘势追击,写完了大部分功能,然后发现了如下缺陷:

  • 这种标记方式本质上,无法应用于下面这样, 没法引用 2-4 这样, 因为会被 # level = 1 的截断;
  • 而且会破坏已有的 ![[file#title]] 的块嵌入.
    boom 💣💣

教训 2

  • 拖了太久,有了思路, 行动比🧠快…

原初 0.3

  • 冷静下梳理需求: 自己的需要仅仅是能在目标区域嵌入多行/段;
    • 多行/段 要么是一个完整的标题, 要么是一个不包含 heading 的纯文本段.
  • 思路确定了: 在 copy-block-link 基础上, 为多行文本添加合适层级的 heading.

不需要做更多工作, 不需要完备, 简单的想法,快速实践, 尽快进入: 开发 - 反馈 - 开发 - 反馈的循环.

  • 这是最符合本人节奏的开发方式, 就不是大局观的那种人, 何必违背自己的性格,患得患失, 反而浪费了很多时间.

Block Link Plus V1.7.5

功能已经比初版复杂很多：

  • 多行块引用 - ^abc123-abc123 格式
  • 时间线聚合 - dataview 查询，提取章节内容
  • 流式编辑器 - 集成了 CodeMirror 6 + React
  • 内联编辑 - !!<a href="/jasper/768ds2323sd/">block-link-plus 开发备忘 & 小记</a> 可编辑嵌入块
  • 别名显示 - 显示内容摘要而不是随机 ID

  • 每个模块职责清晰
  • TypeScript 严格模式
  • 有了 40% 的测试覆盖

一些优化

  • 避免重复渲染：条件检查防止不必要的组件更新
  • React Roots 缓存：管理 DOM 节点生命周期

  • 状态分离：CodeMirror 和 React 各自管理状态

开发历程

开发过程分为几个阶段：

阶段一：基础功能

主要就是搭环境和参考现有插件：

Obsidian Developer Docs 官方文档自然是最开始的入口.

  • 按照 Build a plugin 搭建相关环境
  • 大致浏览 Plugins 内容

之后只需要 Ctrl-C Ctrl-V 不是, 最快的学习自然是已有的开源插件了, 这里开发过程中包括参考了大量插件代码:

  • copy-block-link - 块引用基础逻辑
  • Text Transporter - 多块处理启发
  • Obsidian-Basics - Flow Editor 核心代码移植
  • rendered-block-link-suggestions - 渲染优化思路

小到如何弹出一个通知, 大到如何修改阅读模式渲染内容, 总有一款插件的某一段代码能够参考……

阶段二：架构重构

代码越来越复杂，开始拆分模块：

主要变化：

  • 上了 TypeScript 严格模式
  • React 组件化
  • CodeMirror 6 集成
  • Jest 测试

拆分原则：

  • 每个模块功能单一
  • 减少对 Obsidian API 的直接依赖
  • 加了错误处理

阶段三：复杂功能

Timeline 功能比想象中复杂：

遇到的问题：

  • Dataview API 集成
  • 文档结构解析
  • 用户编辑冲突
  • 大文档性能问题

解决思路：

  • 章节级别的精确匹配
  • 哈希对比避免重复写入
  • React Portals 管理组件生命周期
  • 条件渲染优化性能

工具和环境

调试环境：

  • Ctrl Shift + I obsidian 调试模式, F5 刷新重新加载
  • hot-reload 自动化加载
  • BRAT release 测试

构建发布：

  • esbuild 0.17.3 打包 main.js manifest.json styles.css
  • TypeScript 编译 + 严格类型检查
  • Jest 单元测试 + 40% 覆盖率
  • GitHub Actions 自动化发布流程

AI 工具：

  • Copilot：写代码、debug、测试基本靠它
  • GPT：讨论架构设计，但要小心技术方案的准确性

一些问题和计划

目前的问题

  • Flow Editor 渲染：模式切换时 UI 清理不干净
  • DOM 依赖过重：依赖 Obsidian 内部 DOM 结构
  • 状态管理分散：CodeMirror 和 React 状态没统一

优化方向

  • 统一渲染管理
  • 减少手动事件绑定
  • 考虑轻量状态管理

功能计划

  • 多行块格式增强 (嵌套支持)
  • Timeline 大文档性能优化
  • 完善国际化
  • 标准化插件 API

总结

技术方面的收获

  • TypeScript: 从基础语法到架构设计
  • React: 从简单组件到状态管理
  • 架构思维: 从单文件到模块化系统
  • 测试: 从手动测试到 40% 单元测试覆盖

一些心得

之前的教训依然有效:

  1. 可行性先行: 想法需要技术验证
  2. MVP 思维: 快速验证，迭代优化
  3. 架构演进: 功能复杂了就要重构

新的体会:
4. 模块化价值: 复杂系统下模块化的维护优势很明显
5. 测试投资: 早期测试在后期重构中收益很大
6. 文档重要: 好文档能提升用户接受度
