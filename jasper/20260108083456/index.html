LAB3

lab3

  • TCPSender

这个实验难度陡然上升了不少, 大量 tcp 状态机细节.

  • 逻辑不算复杂, 但是巨量细节堆砌下调试异常头疼

TCPSender

TCPSender 主要工作: ByteStream 中 read 数据, 发送到对端的 receiver.

  • tcp 状态机处理, syn ack fin 等 flag 的发送接收记录.
  • ARQ (Automatic Repeat Request) 当定时器超时, data 没有被对方 ack 确认, 触发重传.

ARQ: 重传超时未确认 data

  • sender 只有一个 timer, 只为第一个未被确认的 seq 定时.
  • 超时发生时, 重发 seq;
  • 当一个 seq 的超时再次发生时, 计数 +1, 下次定时间隔 x2. 外部可获取到这个重试此时绝对是否终止传输.

TCPSender 依赖于 package tcp 具体的调度都是外部触发.

  • 例如真正发送时候, 是从 segmentsOut 中读取 TCPSegment 再走到底层
  • 外部发觉定时器超时了, 调用 Tick 根据重传计数, 决定是否终止传输

主要有 3 个方法

  • FillWindow: 主要的发送入口, 从 stream 读取 data, 构造 TCPSegment, 尽可能填满发送窗口.
    • 包括了 syn fin 等 flag 的发送.
  • AckReceived: 收到对端的 ack 确认, 更新 ack 和 对端窗口大小; 处理 outstanding segments; 最后继续 FillWindow
    • 根据 ackno 清理 outstanding segments 已被确认的 seg.
    • 如果 outstanding segments 不为空, 此时相当于之前的 seg 确认了, 就需要 清理 timer, 准备未下一个 seg 启动定时器了.
    • 如果恰好排空了. 那么非常好 outstanding segments 不再需要定时器了.
  • Tick: 定时器超时了, 触发重传, 增加重传计数和定时间隔

细节:

  • 如果窗口大小为 0 则视为 1: 防止死锁 (有点类似)?, 至少保证窗口 1 还能发送 1 字节的探测包, 不至于 对端一次窗口满了 发送窗口为 0, 下次检查也为 0 , sender 不发送, 接收方没有包收到也不回复, 互相死锁在这了…
  • 代码中对 syn 和 fin 的多次发送做了预防.
  • 我们没有实现 选择确认, 假如 out 中第一个待确认的 ackno 只确认了一半 并不会截取一半发送, 而是重传整个 seq.
    • SACK 选项, 这是一个普遍支持的 tcp 可选配置
