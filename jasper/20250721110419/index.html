记录贴 - GPT 尝试 第五弹

导语

好久没有好好写 blog 了, 自从 6 月搬家后似乎陷入开发泥潭中, 被困住了.

现状

个人项目开发进入了一个泥潭, 严重超期. 落入了名为工具的陷阱中, 最近才重新爬出来…

主要问题:

  • 多次重构, 开发思路出现了 3 次重大转折.
  • AI 工具生产了超量代码, 几乎无法人工审核.
  • 工作流不稳定, 相同模块,相同设计, 每次生产的代码千差万别.

焦头烂额的很长一段时间直到看到了这句话:

在 vibe coding 时代，千万别让工具把自己逼死
我就是让工具把自己逼死了….

总结原因:

  • 个人对设计的把控问题, 推倒 3 次重来,这本身是延期的主因.
  • 过度依赖 vibe coding, 导致人工无法对代码进行任何干预.

解决: AI 只是帮助更快完成, 即使没有 AI 这些步骤也应该能够正常进行.

  • 完全重构工作流, 不再依赖于 AI 的未知性, 而是强制每个步骤都有精确设计, 人或 AI 都能完成 PLAN, 没有额外发挥空间.
  • 不再进行 vibe coding 很遗憾, 现在个人还无法在一个陌生领域开发中驾驭 vibe coding

工作流 -> DPTR-SIGMA

DPTR-SIGMA

这是基于 riper5 和 SamPeng87/tdd-code-implementer 重组而来的 Claude code 的工作流. 还在持续修正中, 主打一个不让 AI 自由发挥.

  • 基于 Claude code subagent

Design loop: CC Plan mode 模块设计 -> subagent 审核 -> 迭代直到完成

  • 架构设计 -> 模块设计 -> 详设

Plan loop: 将详设展开为 tdd loop 可执行计划, 同样是两个 subagent 互相迭代.

  • tdd-loop 非常慢, 引入了 subagent 并行, 目前测试来看一个 cc 会话支持 3 个 subagent 并行是最佳数量.

TDD loop: 复制自 Sampeng87, 严格的 tdd 迭代流程, 基本能保证对相同计划, 产出的代码具备很好一致性.

因为还未彻底完工, 暂时不写更多介绍, 只说几个定制 CC 遇到的几个问题:

CC 多个 Subagent (同一个 Subagent 也可并行) 并行

要求 CC 在一个 message 中有几个 subagent 并行调用, 目前 CC 将一个 subagent 调用 == 调用几次 task , 因此下面是对应 prompt

PARALLEL EXECUTION: For |cycles| instances, invoke Task tool |cycles| times in ONE message. Example: [Ω₃ᵍ×|cycles|] where cycles={1,2} requires 2 Task invocations in the same message.

主线程 / Subagent 通信

默认情况下 subagent 与 CC 是完全独立的上下文, 但是在一个流程中会需要多个 subagent 协调 -> mcp-memory

为一个主题生成一个 node-id -> 所有 subagent 均使用这一个 node-id 检索上下文.

Subagent Edit 调用失败

经常在 subagent 中出现无法编辑文件问题, 但是找不到稳定的复现条件 -> mcp-filesystem

尾巴

在 vibe coding 时代，千万别让工具把自己逼死 这句话最终点醒了我, AI 可以提效, 但是在你身上尚未是真正的 x10 效果, 盲目开坑直接坑底出不来了….
