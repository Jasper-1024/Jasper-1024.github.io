LAB4

lab4

  • TCPConnection

一个最关键的节点, 真正把 sender 和 receiver 结合起来, 跑流量了… 按照 pdf 如果 lab123 完成的不错, 大概只需要 100 多行代码就能实现 lab4 了…

但是吧, lab4 几乎是花费时间最多的时间… 涉及到了 tcp 的状态机的互相转换.

但是说实话, lab4 的状态机与 教科书的 tcp 状态机还是有些区别的

TCP 的状态判断是在 TCPConnection 外部 tcp_state.go, 定义了 tcp 状态机, 而发送/接收端具体状态由 StreamOut StreamIn 来判断

  • receiverStateSummary senderStateSummary 下面会经常看到

TCPConnection

外部的 tcp 保证了正确的调用, TCPConnection 实现细节.

首先最大一个变化, 原来 send recevier 中切片转为了 chan 实现. 原因有 2

  • C++ 队列语法上并不完全对应 go 的切片, 外部拿到队列指针可以修改任意内部. 切片指针只能对底层数组修改, 不能修改切片本身属性.
  • chan 更加的便利, 没有锁,天然的消费者 - 生产者队列. 完成任务上比 c++ 队列更加恰当,都不用关心清理了…

lab4 就啰嗦一点, 重要方法都详细说下.

接下来是比较重要的 SegmentReceived, 其包含了比较复杂的状态转换

另一个重要的 Tick, 相当于给时钟 +1s

  • rto 重传 和 fin-wait

Benchmark && 优化

sponge-go/cmd/tcp-benchmark/main.go 这里与原本稍有不同

  • 融入了 pprof.StartCPUProfile 多了一个采样分析热点的流程 cpu 和 内存信息保存到文件
  • 扩大到 2Gb 数据传输, 还是 100mb 伪随机数据重复 20 次… 100mb 的传输时间太短了, 测量结果忽高忽低…(这里每次测试有 4GB 内存占用,注意)

初始状态下 3.11 / 2.93 Gbit/s

  • 注意: 这个测试结果还是可能波动

热点很多, 有两个明显 ByteStream.Write() 9.8% + 5.8% 和 ByteStream.Peek() 5.4%

ByteStream.Peek()

  • 很明确 创建切片和 copy .

很明确这一处热点已完全消除. 性能 +0.2 Gbit/s +7%

ByteStream.Write

现在再查看火焰图 write 已经占据到了 10.6% + 7 % 最大单点…

咋办? -> 换成 ring 😂, 一次性杜绝 copy 和 mallock

ByteStream.Write 现在只占 5%, 性能 +0.5 Gbit/s +24% 可观的提升.

  • memmove mallocgc 均大幅下降 growslice 完全消除 (没有 pop 的膨胀了)

接下来的热点显然就是 TCPSender.FillWindow ( SetPayload )

TCPSender.FillWindow ( SetPayload )

目前其占据了 6.3%

显然这里需要优化下逻辑而非简单去掉 copy.

查看其他调用 SetPayload 地方, 其实并不需要在 SetPayload 中调用 copy 而且 make + copy 引起的 memmcopy 和 makeslice 本身非常耗资源.

  • peek 这里已经是返回了引用, 最好在 peek 后直接 append
  • SetPayload 直接赋值, 这样编译器或许能直接内联减少一层函数调用.

理论上会减少一层函数调用 和 append 替换 make + copy; 显然这次 " 优化 " 仅有一点效果, 意料之中.

  • 跑几次效果时好时坏, 无法稳定复现.

火焰图上看到 memmove + makesile 被 growslice 替换了, 这次优化只起到了些许作用.

接下来目光投向 PushSubstring

StreamReassembler.PushSubstring

单体占据了 22.8% 的时间…

convT newobject growslice 这 3 个是大头, convT 是 btree 先不动, 看看 newobject 和 growslice

首先来看 data = append([]byte(nil), data…) 似乎不是必要的, 从底层收上来到这里没有别的地方会修改这个切片了,可以去掉 (当然这一点需要外部代码保证).

效果是立竿见影… 双双突破到了 4 Gbit/s; growslice 几乎被完全消除.

显而易见接下来就是 newobject 这里了, 类似 ByteStream 改为 ring 这里也使用资源池思路 sync.Pool

  • 全局预分配资源池, FillWindow 时候 get ack 中 Put
  • 但是从逻辑上看

效果明显, 差不多能到 4.5 G/bits

从火焰图上看, 效果出现在了 AckReceived FillWindow 这两个修改的地方, cpu 时间几乎减半.

接下来的优化就进入深水区了…covnt 和 newobject 还是 7.4% 和 6.5%,

StreamReassembler.PushSubstring

单体还是 22.7% 继续优化, 换支持泛型的 btree

就不堆砌代码了, 只是一个 btree 的替换.

性能终于能跑到 5 Gbit/s, PushSubstring 降低到了 13%, convT 完全消除, newobject 几乎消失.

接下来的优化收益可能就不会这么高了. BytesInFlight 1.5%+3.4% AckReceived 10.5%

其他

TCPSender.BytesInFlight

BytesInFlight 1.5%+3.4% 差不多 5%, 先看看原因.

不遍历也有办法, 只能维护一个变量, 及时增删, 返回时候返回常量

从火焰图上看, BytesInFlight 是完全消失了. 性能上没有太大变化, 似乎更容易跑到 5 Gbit/s 了

Btree Fast Path

PushSubstring 流程中别管其 seq 是否是有序的,每次都会进入 btree 的查询… 其实可以有个快速路径

  • index == assembled 且 btree 为空, 写入 output,跳过所有 btree

效果还是很明显, PushSubstring 的 cpu 时间从 840 降低到了 650ms; 性能也几乎稳定可以跑到 5Gbit/s

再往下优化就很难了…几乎都是必要的 copy …如果哪天时间有闲再说了.

结语

最终 lab4 能跑到 5.81 Gbit/s 远远超乎我的预期, 同时优化过程感觉比 c/c++ 要流畅的多. 有可能 c/c++ 可以优化出更高的上限, 但是 go 的易用… 各中比较下还是开发速度优先可能…
