Clixon 线程模型

官方文档:

“The Clixon programs run as non-blocking single-threaded applications.”

Clixon 是一个单线程应用程序,非阻塞 I/O 和事件驱动模型.

官方文档:

“The callback code must be written with this programming model in mind. The behavior of the callback directly impacts the behavior of the caller and the whole system.”

插件回调必须遵循 Clixon 的单线程设计约束.

Event

定义在 lib/src/clixon_event.c; 是一个小型的 runtime;

  • restconf backend netconf 都是使用这套事件处理;

struct event_data

  • (*e_fn) 是重点

static 全局的事件列表

接口函数

clixon_event_loop 是 event 的核心

单线程事件循环:

  1. select() 监听多个文件描述符
  2. 当有描述符就绪时,依次 *e->e_fn 调用对应的回调函数
  3. 所有回调在同一个线程中顺序执行

Backend

backend_main.c main 是 backend 的入口

  • backend_server_socket 中创建 fd 注册 fd 回调 backend_accept_client ^ec91i
  • 启动 clixon_event_loop

Restconf

restconf_main_native.c 入口

  • 注册的回调函数是 restconf_accept_client
  • 最终也是 clixon_event_loop

RESTCONF Get 处理全流程

当 RESTCONF 请求到达时触发 restconf_accept_client 回调, 建立连接;

一个 Get 请求最终会进入 api_data_get -> api_data_get2, 这个函数将 restconf 转为 netconf 请求

  • 解析 RESTCONF 路径并提取查询参数 将 RESTCONF 路径转换为 XPath
  • 调用 clicon_rpc_get 向 Backend 发送 NETCONF GET 请求

clicon_rpc_get -> clicon_rpc_get2 -> clicon_rpc_msg

clicon_rpc_msg -> clicon_rpc_msg_once -> clicon_rpc_connect

此时请求真正第一次到达了 backend 触发了 backend_accept_client 为这次请求创建 fd 等;

消息处理, client 消息可读了, 执行 from_client -> from_client_msg

from_client_msg -> rpc_callback_call -> from_client_get

from_client_get -> get_common --xx-> get_state_data

get_common–xx-> get_state_data

  • 两个分支: get_list_pagination 和 普通 对于 state data 最终都会进入到 get_state_data

get_state_data -> clixon_plugin_statedata_all -> clixon_plugin_statedata_one

clixon_plugin_statedata_one 最终调用到一个插件 cb 地方

线程模型分析

  1. 单线程执行: 整个 backend 在单个线程中运行,包括事件循环和所有插件回调
  2. 同步调用: 插件回调在主线程中同步执行,如果某个回调执行耗时操作,整个事件循环就会被阻塞
  3. 无并发机制: 没有任何机制允许在一个请求处理期间处理其他请求

总结

Clixon Backend 使用单线程、事件驱动的架构模式,所有操作 (包括 GET 和 SET 请求) 都在同一个主线程中按顺序处理.这种设计简化了内部状态管理,避免了复杂的同步问题,但代价是容易出现性能瓶颈 - 当某个插件回调执行耗时操作时,整个系统都会被阻塞.
